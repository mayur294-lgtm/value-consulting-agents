IT Challenges Blocking These Goals

Hypothesis — for validation
Challenge 1 — blocks Goal 1

Separate Codebases for Web & Mobile

Web and mobile banking are custom-built independently. Every feature is built twice, release cycles are 8-12 weeks apart, and there is no shared component library. This makes omnichannel impossible.

Challenge 2 — blocks Goal 2

Batch-Dependent Core Banking Integration

Several critical data feeds from Temenos T24 (financing eligibility, credit bureau results, compliance status) run on end-of-day batch. Customers and employees are making decisions on yesterday's data.

Challenge 3 — blocks Goal 3

Shariah Compliance is a Disconnected Silo

The Shariah Compliance System operates independently from the financing journey. Product eligibility, profit rate checks, and compliance approvals require separate API calls with no orchestration. Adds 2-5 minutes per interaction.

Challenge 4 — blocks Goal 4

Point-to-Point Integration Spaghetti

18+ direct integrations between systems with no API gateway or event bus. Changes to T24 cascade unpredictably. Each new feature requires touching multiple integration points. Average integration change takes weeks.

Challenge 5 — blocks Goal 5

No Unified Employee Platform

Branch staff, contact center agents, and RMs each use a different combination of 11-13 systems. No single tool provides customer context, digital journey visibility, or guided workflows. $5.4M annual switching tax.

Why These Challenges Exist | Root Cause Hypotheses

Every challenge has a root cause in the architecture. Here are our hypotheses about why DIB faces these challenges. Let's validate together.
H1 | No Experience Platform

Channels were built as standalone projects, not on a shared platform

Web banking was built first (likely 8-10 years ago), then mobile was added as a separate project. Without a shared experience platform, every channel is its own silo with its own tech stack, its own APIs, and its own release cycle.

Root cause of: Challenge 1 (Separate Codebases)
Validation: When was each channel built? By the same team or different vendors?
H2 | No Integration Orchestration Layer

Systems connect directly to each other with no middleware hub

Without an API gateway or integration bus, every system talks directly to every other system. This creates a fragile web where a change to one API breaks three others. New integrations take weeks because each is custom.

Root cause of: Challenge 4 (Integration Spaghetti)
Validation: How does data currently flow between T24 and your channel applications?
H3 | Core Banking as Bottleneck

T24 was designed for batch processing, not real-time digital experiences

Temenos T24 is a strong core banking system, but many of its data feeds are batch-oriented. The original architecture assumed employees would query it during business hours — not that customers would need instant answers 24/7 from a mobile app.

Root cause of: Challenge 2 (Batch Dependency)
Validation: Which T24 operations are real-time vs. batch today?
H4 | Shariah Compliance Bolted On

Compliance was added as a checkpoint, not designed into the journey

The Shariah Compliance System was likely implemented as a separate approval gate — not woven into the product and journey design. This forces a "stop and check" pattern instead of continuous compliance embedded in every step.

Root cause of: Challenge 3 (Shariah as Silo)
Validation: Is Shariah approval a real-time API call or a manual queue?
H5 | Identity Fragmentation

Customer identity lives in 3+ systems with no unified view

T24 has CIF numbers, Salesforce has Contact IDs, channel apps have login credentials. No single identity layer connects them. Employees must search multiple systems. Customers have separate logins per channel.

Root cause of: Challenges 1 + 5 (Channels + Employee Platforms)
Validation: How many separate logins does a DIB customer maintain today?
H6 | No Journey Orchestration

Multi-step processes are hard-coded in each channel instead of centrally managed

Account opening, financing origination, and onboarding flows are coded directly into each channel application. There is no central journey engine that can route steps across channels, embed compliance, and track progress.

Root cause of: Challenges 1 + 2 + 3 (all experience challenges)
Validation: Can a customer start an application on mobile and continue in-branch today?

Discussion

  • Do these root causes ring true? Which ones surprise you?
  • Are there other architectural root causes we should surface?
  • Has DIB attempted to solve any of these before? What happened?
