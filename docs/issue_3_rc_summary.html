#3 ROI Agent Hardcoded to Onboarding/Servicing Critical

7 reinforcing root causes locked the ROI agent into always producing onboarding + servicing levers regardless of engagement context. Fixes introduce an 8-category value lever menu, evidence-driven lever selection, lifecycle coverage checks, and Ignite ingestion for hybrid engagements.

RC1 Hardcoded example biases toward onboarding + servicing roi-business-case-builder.md Implemented

Problem

The embedded JSON config example showed exactly two lever types. The "Servicing Analysis Structure" was a REQUIRED section with no equivalent for retention, revenue growth, or cross-sell.

Fix Applied

B+C Added a Value Lever Reference Menu (8-category table covering Acquire/Activate/Expand/Retain) and Knowledge References section pointing to 5 mandatory reference files. Replaced the hardcoded 2-lever JSON example with a lifecycle-complete config.

Options Not Chosen

A Expand the existing example — add more lever types to the existing JSON snippet. Rejected: doesn't provide the categorical reference structure consultants need.

RC2 All ROI examples are onboarding + servicing heavy roi_examples.md Implemented

Problem

Three worked examples (Seabank, HNB, Goodbody) all led with onboarding. Categories 4 (Loyalty & Retention) and 5 (IT Cost Savings) had zero templates.

Fix Applied

A+B Added Example 4: MSB Vietnam (retention $1.69M, cross-sell $4.77M, CC origination $3.60M). Added Example 5: Wealth composite (RM productivity $24.6M dominant, onboarding $186K minor). Expanded Category 4 & 5 templates.

Options Not Chosen

C Rebalance existing examples — restructure Seabank/HNB/Goodbody to lead with non-onboarding levers. Rejected: changing historical examples would reduce credibility; better to add new ones.

RC3 JSON config schema enforces two journeys roi_config.json (retail + wealth) Implemented

Problem

Config had a hardcoded journeys object with exactly customer_onboarding + customer_servicing.

Fix Applied

C3 Renamed journeys → value_lever_groups with category and lifecycle_stage tags. Created wealth config with 5 lever groups where RM productivity dominates (73%).

Options Not Chosen

A Make journeys a dynamic array — replace the named object with an array of journey objects. Rejected: loses the semantic naming that makes configs readable.

B Add a value_levers flat array — alongside journeys. Rejected: duplicates structure, creates sync issues.

C1 Simple rename only — rename journeys without adding category/lifecycle tags. Rejected: misses the opportunity to embed lifecycle semantics.

C2 Rename + flatten inner structure — simplify the per-group schema. Rejected: loses the rich input/evidence structure that makes configs auditable.

Agent went straight from "read inputs" → "build model" with no intermediate step to inventory evidence, map lifecycle coverage, or validate lever selection with the consultant.

A Separate "Value Lever Discovery" phase — add a new section before the checkpoint. Rejected: creates two sections doing related work; better to merge into one enriched checkpoint.

C Two-stage checkpoint pattern — split into Checkpoint 1 (scope) and Checkpoint 2 (numbers). Rejected: too heavyweight; most engagements don't need two full stops.

Fix Applied

Options Not Chosen

A Enrich handoff entry only — expand the 3-line handoff without updating the data flow diagram. Rejected: the diagram would be out of sync with the handoff spec.

B Add a "Handoff Brief" template — define a structured brief the orchestrator must prepare. Rejected: adds orchestrator complexity without proportional benefit; the enriched handoff entry is sufficient.

#4 Excel ROI Model — Full Formula Rewrite Critical

The Excel generated by roi_excel_generator.py wrote hardcoded static values into every cell. No formulas, no cross-sheet references, no named ranges. Changing an input recalculated nothing. Complete rewrite to a formula-based architecture with scenario switching.

RC1 Python computes everything, Excel stores static numbers roi_excel_generator.py Implemented

Fix Applied

Full rewrite (~1,300 lines). All lever benefits, servicing savings, cashflow totals, NPV, IRR, and payback period are now Excel formulas. Python only writes the structural config values to input cells; all downstream cells are =references.

Options Not Chosen

A Partial formula — only Cashflows/Dashboard — keep lever calculations in Python, only make rollup sheets formula-based. Rejected: inconsistent model; changing a lever input still wouldn't recalculate the lever sheet itself.

B VBA macros for recalculation — add VBA that recalculates on input change. Rejected: VBA introduces security warnings, macro-enabled format (.xlsm), and compatibility issues across Excel/Numbers/Sheets.

RC2 No cell reference architecture roi_excel_generator.py Implemented

Fix Applied

Introduced self.cell_map dict tracking dynamic cell positions across sheets. Named ranges for global inputs: ScenarioIndex, DiscountRate, License_Y1..Y5, Impl_Y1..Y5. Cross-sheet references use the cell_map for dynamic formula construction.

Options Not Chosen

A Hardcoded row numbers — use fixed row/column constants for cross-sheet references. Rejected: breaks when lever groups vary in size between engagements; cell_map adapts dynamically.

B Named ranges for everything — create named ranges for every single cell reference. Rejected: would produce hundreds of named ranges making the workbook hard to debug; cell_map + named ranges for globals is the right balance.

RC3 No Input → Calculation → Output sheet architecture roi_excel_generator.py Implemented

Fix Applied

New sheet architecture: Scenario Data (3-column reference) → Model Inputs (dropdown + INDEX formulas) → Lever Sheets (one per group) → Cashflows (cross-sheet formulas, =NPV(), =IRR()) → Results Dashboard (all references). Scenario switching via DataValidation dropdown (1/2/3) — no VBA.

Options Not Chosen

A Flat sheet structure — keep all calculations on one sheet with inputs at top. Rejected: unmanageable for engagements with 5+ lever groups; per-sheet architecture scales and isolates concerns.

B Multiple workbook files — separate input workbook from calculation workbook. Rejected: cross-workbook references are fragile and break when files are moved or emailed.

Prompt ROI agent prompt updated for formula-based model roi-business-case-builder.md Implemented

Fix Applied

Updated "Excel Model Structure" section with data flow diagram, sheet descriptions, scenario switching mechanics, named ranges, and value_lever_groups config format with full JSON example.

Bugs 3 bugs found and fixed during testing roi_excel_generator.py Implemented

Fixes Applied

  • Double spaces in sheet names: _safe_sheet_name stripped em-dashes but left double spaces. Fixed with re.sub(r'\s+', ' ', clean).
  • No grand total in Servicing Detail: Channel subtotals existed but no overall total. Added channel_subtotal_cells tracking and grand total row.
  • Servicing total hardcoded on lever sheet: Changed from static value to ='Servicing Detail'!{grand_total_ref}. Required reordering sheet creation (Servicing Detail before lever sheets).

B Embed market research into ROI agent — have the ROI agent do its own market research. Rejected: violates single-responsibility; the Market Context Researcher is a specialized agent for this purpose.

RC3 No peer bank confirmation before research market-context-researcher.md Implemented

Fix Applied

Added Step 3b: peer bank confirmation checkpoint. Agent presents proposed peer banks to consultant and waits for approval before researching.

Options Not Chosen

B Skip validation, review after research — let the agent research all proposed peers, then have the consultant review. Rejected: wastes time researching irrelevant peers; cheaper to confirm upfront.

#6 ROI Calibration Against Benchmarks High

The ROI agent blindly executed when results fell short of domain benchmarks. No calibration check, no suggestion of expansion levers or parameter adjustment.

RC1 No benchmark comparison step roi-business-case-builder.md Implemented

Fix Applied

Added Section 6 "ROI Calibration Check (MANDATORY)" with domain benchmark table (Retail 60-150%, Wealth 120-200%, Commercial 80-140%). Agent compares computed ROI against benchmarks and presents 4 options to consultant: proceed, add levers, adjust parameters, or flag as realistic. Updated quality checklist with calibration items. Renumbered subsequent sections.

Options Not Chosen

A Auto-calibrate — if ROI falls below benchmark, automatically add expansion levers or adjust parameters upward. Rejected: artificially inflating ROI to hit a target undermines model credibility; the consultant must decide how to respond.

B Warning-only — just flag the shortfall in the report without presenting options. Rejected: flagging without actionable options leaves the consultant without a path forward; the 4-option decision prompt is more useful.

#7 Domain-Aware Ramp-Up Curves High

Implementation/adoption curves were static — same curve for a 6-month retail deployment and a 24-month wealth transformation. Curves should derive from domain, timeline, and complexity.

RC1 Static curves — no domain/timeline/complexity awareness ramp_up_models.md (new) Implemented

Fix Applied

Created knowledge/standards/ramp_up_models.md with 4-step derivation: (1) Estimate timeline from domain/journeys/integration/org readiness, (2) Look up implementation curve from timeline table (6mo to 24mo), (3) Apply domain-specific adoption multiplier (retail self-service 0.85x, wealth 0.50x Y1, commercial 0.45x Y1), (4) Apply scenario variation. Includes complexity multipliers (multi-country +6mo, regulatory +3-6mo, core migration +6-12mo). Updated ROI agent to reference this file instead of static table.

Options Not Chosen

A Embed curves directly in ROI agent prompt — inline the full derivation tables into the agent markdown. Rejected: would add ~80 lines to an already long prompt; a separate knowledge file is cleaner and reusable by other agents.

B Code-based curve generator in Python — build a tools/curve_generator.py that computes curves programmatically. Rejected: over-engineered for a prompt-based agent system; the agent reads a table and interpolates, no code needed.

#8 Growth-Adjusted Servicing + FTE Avoidance Medium

Servicing lever only captured point-in-time savings, ignoring the growing cost of NOT automating. For a bank growing 8% YoY, the cost of manual handling compounds — digital avoids hiring additional staff.

RC1 No future cost avoidance from headcount growth roi-business-case-builder.md + roi_excel_generator.py Implemented

Fix Applied

Agent prompt: Added growth-adjusted cost avoidance template with formulas: Growth Volume = Base × ((1+rate)^year - 1), FTEs Avoided = Digital Handles × Avg Time / 1,800 hrs. Added FTE equivalence guidance for executive communication.

Excel generator: Added ~80 lines for growth-adjusted section in Servicing Detail (YoY volume growth, digital handling, FTEs avoided, cost avoided — all as Excel formulas). Wired growth avoidance into Cashflows as additional inflow row. Fixed column alignment bug (col = 4 + yr → col = 3 + yr). Handles both yoy_growth array and yoy_growth_rate scalar config formats.

Options Not Chosen

A Prompt-only (no Excel change) — describe growth adjustment in the agent prompt template and let the agent compute values before writing to Excel. Rejected: defeats the Issue #4 formula-based architecture; growth projections must be live Excel formulas so consultants can adjust the growth rate.

B Separate "Growth Analysis" sheet — create a dedicated sheet for growth projections. Rejected: over-fragments the model; growth avoidance is a sub-section of servicing, not a separate concern. Keeping it in Servicing Detail maintains the logical grouping.
